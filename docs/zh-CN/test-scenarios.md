# WebBrain 测试场景

一组端到端任务，用于评估 WebBrain 浏览器代理在不同难度和 UI 模式覆盖范围下的表现。每个场景列出了站点、要粘贴到侧面板的提示以及可观察的通过标准。

## 场景

### 1. Wikipedia 摘要 —— 简单
- **站点：** en.wikipedia.org
- **任务：** "去 Wikipedia，找到关于 Alan Turing 的文章，告诉我他的逝世日期。"
- **预期：** 代理导航，阅读信息框，回答"1954 年 6 月 7 日"。一次 `get_accessibility_tree` → `done`。

### 2. Hacker News 头条故事 —— 简单
- **站点：** news.ycombinator.com
- **任务：** "Hacker News 上现在的 #1 故事是什么，它有多少分？"
- **预期：** 从顶部行返回标题 + 分数。无需点击。

### 3. Amazon 价格查询 —— 简单
- **站点：** amazon.com
- **任务：** "在 Amazon 上搜索 'USB-C hub'，告诉我顶部赞助结果的价格。"
- **预期：** 在搜索框中输入一次，按一次回车，阅读第一个卡片。测试搜索框检测和 SERP 解析。

### 4. GitHub 问题提交 —— 中等
- **站点：** github.com（你自己的测试仓库）
- **任务：** "在 github.com/<你>/<仓库> 上打开一个新 issue，标题为 'Test from WebBrain'，内容为 'ignore this'。"
- **预期：** 代理导航到 `/issues/new`，填写标题 + 内容，点击提交。测试 contenteditable 内容编辑器和带有重复标签的"Submit new issue"按钮。

### 5. Gmail 撰写草稿 —— 中等
- **站点：** mail.google.com
- **任务：** "撰写一封草稿邮件给 test@example.com，主题为 'hello'，内容为 'this is a test'。不要发送。"
- **预期：** 点击"撰写"，填写三个字段（收件人带自动完成），关闭以保存为草稿。测试 React 门户和自动完成组合框。

### 6. Google 日历事件 —— 中等
- **站点：** calendar.google.com
- **任务：** "创建一个日历事件，标题为 'Lunch'，明天中午 12:30，持续 30 分钟。"
- **预期：** 打开创建对话框，输入标题，调整日期/时间选择器，保存。测试日期/时间微调器。

### 7. Stripe 产品（CNY 定期） —— 困难
- **站点：** dashboard.stripe.com
- **任务：** "创建一个名为 'MyProduct' 的产品，定价 500 人民币，每 2 个月定期扣费。"
- **预期：** 产品目录中有一行，名称为 MyProduct，货币为 CNY，间隔为 2 个月。测试虚拟化组合框 + 自定义间隔显示。

### 8. LinkedIn 连接请求 —— 中等
- **站点：** linkedin.com
- **任务：** "访问 linkedin.com/in/<个人资料>，点击 Connect，添加备注 'Met at Demo Day'，然后发送。"
- **预期：** 处理 Connect 被隐藏的个人资料上的"更多"溢出菜单。测试菜单发现。

### 9. Airbnb 搜索筛选 —— 困难
- **站点：** airbnb.com
- **任务：** "查找伊斯坦布尔 6 月 10–15 日的房源，2 位房客，最高 $150/晚，带泳池。"
- **预期：** 填写目的地自动完成、日期范围选择器、房客步进器、价格滑块、便利设施复选框。测试范围输入和多步骤筛选抽屉。

### 10. Google Flights 搜索 —— 困难
- **站点：** google.com/travel/flights
- **任务：** "搜索单程航班 SFO → IST，2026 年 5 月 20 日出发，经济舱，1 位成人。返回最便宜的选择及其航空公司。"
- **预期：** 机场自动完成（两个组合框）、日期选择器、舱位选择器，读取第一个结果卡片。测试 SPA 路由切换。

### 11. Reddit 子版块订阅 —— 中等
- **站点：** reddit.com
- **任务：** "让我订阅 r/chess，然后告诉我它有多少成员。"
- **预期：** 点击 Join，从侧边栏读取成员数量。需要已登录账户；测试状态变化检测（Join → Joined）。

### 12. 多标签页研究综合 —— 困难
- **站点：** arxiv.org + scholar.google.com
- **任务：** "打开 arxiv.org/abs/1706.03762，然后在 Google Scholar 上搜索它的引用次数，告诉我论文标题和引用数。"
- **预期：** 使用 `tabs_create` / `switch_browser` 在两个标签页之间切换并返回两个数据点。测试跨标签页状态处理。

## 评分

- **通过：** 正确的答案或正确的最终页面状态。
- **部分通过：** 数据正确但字段错误，或提交的表单有一个字段错误。
- **失败：** 陷入循环、修改了错误的实体或幻觉完成。

## 自动化选项

按从最简单到最宏大的顺序排列。

### 1. 你已经拥有的追踪（最低成本基线）
追踪已经内置。在设置中打开"录制追踪"，为每个要比较的模型手动运行每个场景一次，然后打开追踪页面并排对比。这就是我们本轮一直在做的。启动成本低，但评分是手动的。

### 2. 从侧面板运行场景运行器
添加一个 `tests.json`，包含 `{ url, prompt, check }` 行。构建一个小型测试运行器标签页（镜像追踪），循环遍历条目：对于每个条目，打开 URL，将提示填入侧面板的输入框，等待代理完成，然后评估 `check`。三种有用的 `check` 类型：

- **URL 匹配** —— 最终的 `window.location.href` 匹配一个正则表达式。适用于捕捉"草稿是否真的被创建了？"这类在保存时重定向的页面。
- **DOM 断言** —— 在页面中运行一小段 JS 代码（`document.querySelector('[data-testid=product-row-name]')?.textContent === 'namaz'`）。适用于场景 4、7、11。
- **LLM 作为裁判** —— 将最终的 `done` 摘要 + 截图提供给一个更强的模型（Claude Sonnet），附带提示 + 评分标准，得到通过/部分通过/失败。适用于场景 1、2、3、10、12，其中答案是一个字符串。

你已经拥有所需的一切：代理循环、消息总线、截图、追踪。大约 1 天的工作量。

### 3. 通过 Puppeteer / Playwright 进行无头运行
启动 Chromium 并预装扩展（`--load-extension=path/to/webbrain`），通过其 DOM 驱动侧面板，并非交互式地运行测试套件——可以过夜运行、在 CI 中运行、跨 N 个提供商运行。你可以跨 Chrome 配置文件进行并行化以提高吞吐量。缺点：需要登录的场景（Gmail、Reddit、LinkedIn）需要每个配置文件预置 cookie，而 Stripe 的仪表板对新浏览器很敏感（验证码）。

最佳方案是混合模式：匿名站点（Wikipedia、HN、Amazon、arxiv）使用无头模式，需要登录的站点使用侧面板运行器。

### 4. 从真实浏览生成场景
录制人类完成任务的过程（打开开发者工具），保存点击的元素 ID + 最终的 DOM 状态，并自动构建 `check`。这将每个手动演示的流程转化为回归测试。前期成本较高；一旦你拥有 50+ 个场景，回报就来了。

### 5. 模型 vs 模型排行榜
一旦 (2) 和 (3) 到位，每晚在所有配置的提供商（llama.cpp 4B/12B/31B、Anthropic、OpenAI、OpenRouter）上运行相同的测试套件。将分数持久化到 IndexedDB，渲染一个排行榜标签页。这就是追踪发挥作用的地方——你可以点击"为什么 Gemma-4-31B 在 Stripe 上失败了？"并看到确切的步骤序列，无需重新运行。

### 建议
先构建选项 2。追踪基础设施意味着你已经完成了大约 60%——缺少的是脚本化循环和一个微小的通过/失败裁判。这能给你可重复的量化数据来证明像 `v3.6.8` 这样的更改。当测试套件增长到超过约 20 个场景且每次运行的成本变得重要时，再添加选项 3。
